# GitHub

REST, GraphQL, Check, comment, review, and workflow-dispatch adapters.

`fetchPullRequestPages` is the shared bounded page-number primitive. It supports synchronous and asynchronous adapters, stops after an empty or partial page, and retains the production limit of 30 pages with 100 items per page.

`fetchGitHubLinkPages` follows GitHub `Link` headers within the original API origin, rejects cycles or malformed page payloads, and uses the same 30-page ceiling. API adapters remain responsible for authentication and transport.

`createGitHubRestTransport` accepts an explicit token and an injectable `fetch` implementation. It confines requests to one HTTPS API base URL, exposes that normalized boundary to response validators, writes only bounded GitHub error metadata, and never reads process environment. GitHub.com and the official `api.SUBDOMAIN.ghe.com` GitHub Enterprise Cloud shape default to REST `2026-03-10`; self-hosted GitHub Enterprise Server and unrecognized hosts default conservatively to `2022-11-28` because GHES date-version support follows the server release. A caller may provide an explicit supported API version after capability discovery.

`GitHubRepositoryClient` is the shared integration adapter for the authenticated user, default-branch Manifest loader, bounded PR/Check/workflow/review-thread reads, and one-call/one-mutation primitives. Classification primitives expose repository-label lookup/creation, PR issue-label add/remove, and PR body update without deciding which labels or hidden metadata a project needs. GitHub.com callers can use one transport; GitHub Enterprise Server callers pass separately confined REST (`https://HOSTNAME/api/v3/`) and GraphQL (`https://HOSTNAME/api/`) transports so `/graphql` resolves to the platform's distinct GraphQL endpoint. Closed-PR merge state comes from GraphQL `merged` plus `mergeCommit`, never the removed REST `merge_commit_sha`. Workflow dispatch returns an identified run on the 2026 API and validates both response URLs against the configured REST base and its exact Cloud, GHEC-tenant, or GHES web origin; compatible GHES APIs retain an explicit accepted-without-ID outcome, preventing both lost associations and duplicate retries after a successful legacy 204. Release uploads likewise accept only the canonical Cloud/GHEC upload host or the GHES `/api/uploads/repos/...` path for the configured repository. The client does not decide policy, retry in time, poll for convergence, or choose which mutations to execute; Action operations use core plans to make those choices.
